## Development Sequence For Implementation

1. **Step 1:** Create the file layout under `code/` exactly as planned: `main.py`, `config.py`, `schemas.py`, `corpus.py`, `retriever.py`, `safety.py`, `taxonomy.py`, `llm.py`, `agent.py`, `evaluate_sample.py`, and `README.md`. Do this first so later work lands in the right place and no one improvises structure mid-implementation.
2. **Step 2:** Define config and schema boundaries before writing logic. In `config.py`, define repository-relative paths, the `support_tickets` versus `support_issues` fallback, cache locations, output paths, and provider settings. In `schemas.py`, define the enums and Pydantic models for input tickets, retrieved chunks, safety outcomes, and final outputs. This gives every later module a stable contract.
3. **Step 3:** Implement sample-header inspection and CSV writing early in `main.py`. Read the sample CSV header, store the exact output column order, and make the writer mirror it. Add a startup failure if the sample file is missing or unreadable. This prevents late surprises in the output format.
4. **Step 4:** Implement the simplest end-to-end CLI skeleton in `main.py` and `agent.py` before building real intelligence. The CLI should resolve paths, load the input CSV, iterate rows, call a placeholder agent flow, and write a structurally valid output CSV. At this point the content can be temporary placeholders, but the batch contract must work.
5. **Step 5:** Build `taxonomy.py` next, before retrieval. Seed `product_area` values from the sample CSV, then extend them using corpus breadcrumbs and folder names. Expose helpers for validating emitted `product_area` values and for mapping corpus evidence back to allowed labels. This prevents downstream modules from inventing categories.
6. **Step 6:** Build `corpus.py` to scan markdown files, parse frontmatter, normalize text, and mark stub pages. Keep the first version simple and file-level. Only after file-level parsing works should heading-aware chunking be added for long documents.
7. **Step 7:** Add heading-aware chunking in `corpus.py` only for long files. Use headings and subheadings to split long guides and release notes, but leave short FAQ articles intact. Save the normalized result as a local cache artifact and make later runs reuse it.
8. **Step 8:** Implement `retriever.py` with lexical retrieval and domain filtering only. Start with TF-IDF or BM25 over normalized records or chunks, and require the retriever to return provenance fields such as source path, title, breadcrumbs, heading context, and rank score. Do not add semantic retrieval at this stage.
9. **Step 9:** Implement domain detection in `agent.py` using `company` first and retrieval-backed fallback second. If `company` is present and plausible, trust it. If `company` is `None` or weak, infer the domain using keywords and shallow retrieval against titles and breadcrumbs. If the domain remains unclear, route into escalation instead of guesswork.
10. **Step 10:** Implement `safety.py` before answer generation. Add the concrete escalation categories already defined in the plan and make them deterministic. Also add `request_type` heuristics here so that the system can classify tickets even when answer generation is not ready yet.
11. **Step 11:** Connect the deterministic baseline in `agent.py`. The agent flow at this stage should be: normalize ticket, detect domain, run safety rules, retrieve evidence if allowed, map `product_area` from retrieved evidence, and produce either a fixed escalation response or a basic deterministic response assembled from retrieved support text. This creates a full baseline without depending on an LLM.
12. **Step 12:** Add `evaluate_sample.py` immediately after the deterministic baseline is working. Run the sample set end to end and compare categorical outputs first: `status`, `request_type`, and `product_area`. Use this to identify whether errors are mostly routing, safety, taxonomy, or retrieval issues.
13. **Step 13:** Tighten the deterministic baseline before introducing any provider calls. Fix wrong `product_area` mapping, weak domain routing, missing escalation cases, and retrieval quality problems first. The goal is to make the non-LLM path safe and structurally correct before adding a model.
14. **Step 14:** Add `llm.py` only after the baseline is stable. The wrapper should read environment variables, use temperature 0, request strict JSON output, and expose one bounded call for response synthesis or rare tie-break decisions. Include one retry for malformed JSON, then fail closed.
15. **Step 15:** Use the LLM only in narrow places. The default plan is to use it for bounded response synthesis on safe, well-retrieved tickets and only optionally for difficult tie-breaks. Core routing, safety, taxonomy validation, and retrieval must remain functional without it.
16. **Step 16:** Add fallback behavior in `agent.py` for missing keys, provider failures, malformed JSON, or low-confidence evidence. The fallback path should never block batch execution. It should degrade to deterministic escalation or a safe deterministic response depending on the case.
17. **Step 17:** Rerun `evaluate_sample.py` after adding the LLM and confirm there is no regression in `status`, `request_type`, or `product_area`. If the LLM improves response quality but harms categorical consistency, keep the deterministic path in control and narrow model usage further.
18. **Step 18:** Add a short manual regression set for the likely scoring traps: service-down issues, score disputes, account restoration, Claude conversation deletion versus account deletion, Visa stolen or unauthorized instrument cases, merchant disputes, and obviously unrelated prompts. Use these to verify the safety gate and escalation templates.
19. **Step 19:** Only consider extra retrieval sophistication after measuring actual misses. If lexical retrieval plus chunking fails on clear paraphrase cases, add semantic reranking over the top lexical candidates only. Do not introduce a dense retrieval system, vector database, or separate retrieval engines unless the failure pattern clearly justifies it.
20. **Step 20:** Write code `README.md` once the architecture is stable enough to describe clearly. Include install steps, environment setup, how to run the batch job, how to run sample evaluation, architecture summary, limitations, and the submission checklist including confirmation of the external transcript artifact.
21. **Step 21:** Final hardening pass. Rebuild the cache from scratch once, rerun the sample evaluation, run the production CSV, verify the output header matches the sample exactly, verify all categorical values are valid, and rerun the same command to confirm reproducibility.

## Execution Rules For Implementation Agents

* Do not add the LLM before the deterministic baseline can process the full CSV and write a valid output file.
* Do not add semantic retrieval before sample evaluation shows concrete lexical-retrieval misses.
* Do not optimize wording before `status`, `request_type`, `product_area`, and safety behavior are stable.
* When in doubt, escalate rather than guess.
* Keep every module small and single-purpose; avoid merging safety, retrieval, and taxonomy logic into one file.
* After each major step, run the cheapest verification that proves the new layer works before moving on.

## Milestone Gates

* **Milestone A:** Batch skeleton works. `code/main.py` can read input rows and write a correctly structured output CSV.
* **Milestone B:** Corpus and taxonomy work. The corpus cache builds successfully and `product_area` values can be validated against the allowed vocabulary.
* **Milestone C:** Deterministic baseline works. The agent can route, escalate, retrieve, and emit complete outputs without any provider dependency.
* **Milestone D:** Sample evaluation is informative. `evaluate_sample.py` identifies the main failure categories clearly enough to guide tuning.
* **Milestone E:** Optional LLM layer is additive. Provider-backed synthesis improves safe reply quality without weakening categorical consistency or safety.
* **Milestone F:** Submission-ready run works. The production CSV completes successfully, repeats reproducibly, and the submission checklist is satisfied.
